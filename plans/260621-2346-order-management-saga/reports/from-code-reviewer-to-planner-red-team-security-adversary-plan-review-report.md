## Finding 1: Mock-payment force endpoints have ZERO auth spec → anyone can mark any order paid

- **Severity:** Critical
- **Location:** Phase 6, "Requirements → Control endpoints" and "Implementation Steps step 3"
- **Flaw:** `POST /mock-payments/:id/succeed` and `/fail` are specified only as "force outcomes" (phase-06 line 22, 48). No `preHandler`, no `requireRole`, no `app.authenticate`, no IDOR ownership check is anywhere in the plan. By contrast phase-02 explicitly says products mutations get `requireRole('admin')` (phase-02 line 22) — the omission in phase-06 is conspicuous, not implied.
- **Failure scenario:** Attacker enumerates/guesses a `payment id` (or `order id`) and calls `POST /mock-payments/{id}/succeed`. The webhook side-effect chain fires: payment→paid, reservation committed, `OrderPaid` emitted, shipment created, goods ship. Full payment bypass — order fulfilled without paying. Even if these are "demo only" endpoints, they execute the real saga commit path (phase-06 line 20-21). In a portfolio repo deployed publicly this is a free-checkout button.
- **Evidence:** `plans/.../phase-06-payment-saga-webhook.md:22` ("Control endpoints: `POST /mock-payments/:id/succeed`...") with no auth; contrast `phase-02-product-catalog-cache.md:22` which DOES specify `requireRole('admin')`. Codebase has no role system at all: `grep -rn "role" src/` returns nothing; `src/types/fastify.d.ts:17` JWT user is only `{ sub, email }`.
- **Suggested fix:** Phase 6 must state these endpoints require `requireRole('admin')` AND are disabled when `NODE_ENV==='production'` (env-gate). Add a test asserting customer token → 403 and prod → 404.

## Finding 2: Webhook eventId is attacker-controlled → HMAC dedup defeats neither replay nor forgery as claimed

- **Severity:** Critical
- **Location:** Phase 6, "Architecture → Webhook idempotency" and "Non-functional"
- **Flaw:** Dedup is `SETNX processed:webhook:{eventId}` (phase-06 line 29) where `eventId` comes from the webhook body, which is generated by the mock provider (phase-06 line 28). The signature is `HMAC_SHA256(secret, rawBody)` (phase-06 line 27). The plan treats "valid HMAC + eventId dedup" as sufficient, but never pins what is signed beyond rawBody, and never binds eventId to a server-issued nonce.
  - **Replay:** A valid signed webhook captured off the wire (or simply re-sent by anyone who can read one legit request) replays fine the FIRST time after the TTL expires (phase-06 line 29 says TTL, not permanent). After `processed:webhook:{eventId}` TTL lapses, the identical signed body replays and re-triggers the side effect. No timestamp/expiry is signed.
  - **Forgery via omitted eventId:** Plan never specifies behavior when `eventId` is missing/empty. `SETNX processed:webhook:` (empty) collides across all such requests or NPEs — undefined. A signed body with no eventId field is unhandled.
- **Failure scenario:** (a) Attacker who obtains the `WEBHOOK_HMAC_SECRET` (see Finding 3) forges any payload with a fresh random `eventId`, dedup never fires, marks arbitrary orders paid. (b) Even without the secret, a captured legit success webhook is replayed after TTL → double-commit / re-fulfillment of an already-cancelled order if status guard is weak.
- **Evidence:** `phase-06-payment-saga-webhook.md:19,27,29`. Provider self-generates the call: line 28 "a self-call to the webhook with a signed `SUCCEEDED` payload". No nonce table, no signed timestamp, no eventId-must-exist validation in plan. Existing consumer dedup uses a permanent PK row (`processed_messages.messageId` PK, `src/infra/db/schema.ts:52-55`) — the webhook plan downgrades to a TTL'd Redis key, which is strictly weaker for replay.
- **Suggested fix:** Sign a payload that includes a server-issued payment reference + ISO timestamp; reject if timestamp skew > N minutes (kills replay independent of TTL). Make eventId presence mandatory (400 if absent). Persist dedup in Postgres (permanent) like `processed_messages`, not a TTL'd Redis key, OR set TTL well beyond any practical replay window and document it.

## Finding 3: WEBHOOK_HMAC_SECRET has no strength floor and the mock provider holds it in-process → trust boundary is fictional

- **Severity:** High
- **Location:** Phase 1, "Architecture → New env"; Phase 6, "Architecture → Mock provider"
- **Flaw:** `WEBHOOK_HMAC_SECRET` is declared in env schema (phase-01 line 20, 39) but the plan sets no `minLength`. The existing schema enforces `JWT_SECRET: Type.String({ minLength: 32 })` (`src/config/env-schema.ts:19`) — so a precedent for a floor exists and is silently dropped for the HMAC secret. Worse: the "provider" is in-process (phase-06 line 28) and signs with the same secret the verifier uses. There is no real cross-service trust boundary; the HMAC is theater. If `.env.example` ships a weak/placeholder secret (phase-01 line 39 edits `.env.example`) and it's deployed unchanged, the secret is public.
- **Failure scenario:** Deploy with the example secret (common in portfolio deploys) → attacker reads `.env.example` in the public repo, computes valid `X-Signature` for any forged payload, drives Finding 1/2 without needing the force endpoints at all. A short secret (e.g. 8 chars) is also brute-forceable offline given one captured (body, signature) pair.
- **Evidence:** `phase-01-...md:20` ("`WEBHOOK_HMAC_SECRET` (used phase 6, declare now)") — no constraint stated; compare `src/config/env-schema.ts:19` JWT floor. `phase-06-...md:28` provider is in-process self-call. No `.env.example` secret-generation note in plan.
- **Suggested fix:** `WEBHOOK_HMAC_SECRET: Type.String({ minLength: 32 })`; `.env.example` must carry an obviously-fake value plus a comment "generate with openssl rand -hex 32"; add a boot check refusing the known example value in production.

## Finding 4: JWT role claim is trusted with no spec for token re-issuance or stale-role handling → privilege escalation surface

- **Severity:** High
- **Location:** Phase 1, "Architecture → RBAC"
- **Flaw:** Plan adds `users.role` and reads `request.user.role` from the JWT (phase-01 line 24). But: (a) `signToken` currently signs only `{ sub, email }` (`src/modules/auth/auth-service.ts:7,31`; `src/types/fastify.d.ts:18-19`). The plan says "JWT payload + sign updated to include role" but does not say role is re-read from DB on privileged calls. (b) Tokens live 15m default and longer is configurable (`src/config/env-schema.ts:20` `JWT_EXPIRES_IN`). A user promoted/demoted keeps their old role until token expiry. (c) No spec forbids the client from supplying role — must confirm role is taken ONLY from verified token claims, never request body/query (the orders controller already does this correctly for userId: `src/modules/orders/orders-controller.ts:14-16` uses `req.user.sub`, good precedent, but the plan doesn't restate the rule for role).
- **Failure scenario:** Admin role revoked after compromise; attacker's existing token still passes `requireRole('admin')` for up to `JWT_EXPIRES_IN`. If any code path ever merges body into the token-derived user (a refactor risk the plan doesn't guard), a customer self-asserts `role:admin`.
- **Evidence:** `src/modules/auth/auth-service.ts:31` `signToken({ sub, email })` — no role today; `src/types/fastify.d.ts:18` payload shape lacks role; `phase-01-...md:24` adds role to JWT but no DB-recheck or revocation note. `JWT_EXPIRES_IN` default `15m` at `src/config/env-schema.ts:20`.
- **Suggested fix:** For admin-only mutations, re-read role from DB (cheap, one indexed lookup) rather than trusting a possibly-stale claim, OR document accepted staleness window. Explicitly state role is sourced only from verified claims. Add a negative test: token with hand-crafted `role:admin` but unsigned/resigned with wrong key → 401.

## Finding 5: Idempotency replay returns the stored response body cross-request with no key↔user re-binding check → data leak / cross-user replay

- **Severity:** High
- **Location:** Phase 5, "Architecture" + "Requirements"
- **Flaw:** Key is `idem:{userId}:{routeId}:{Idempotency-Key}` (phase-05 line 20) and the stored `{status, body}` is replayed on hit (phase-05 line 20). userId in the key mitigates cross-user collision IF userId is always present and trusted. But: the plan enables idempotency generically ("other mutating POSTs", phase-05 line 16) and later wants it on "payment webhook"? No — line 50 says webhook has its own dedup. Still, the stored-response replay has two gaps: (a) no verification that the replaying caller's userId equals the stored owner beyond the key prefix — fine if key derivation is bulletproof, but the plan never says the body is filtered/owner-checked before replay; (b) the marker uses `SETNX` then "store on onSend" (phase-05 line 20, 35), creating a window where a crash between SETNX and onSend leaves a permanent "processing" marker → every retry gets `409 retry-later` forever (poison key) since the completed record never lands.
- **Failure scenario:** (a) Request A (user X) succeeds, body stored. Bug/refactor where routeId collides or userId missing (e.g. an unauthenticated route accidentally opted-in) → user Y with same Idempotency-Key replays X's order body (contains X's orderId, totals). (b) Handler crashes post-SETNX, pre-onSend: client retries forever, all 409 — self-inflicted DoS on that key for the full 24h TTL.
- **Evidence:** `phase-05-...md:17,20`. SETNX-then-onSend ordering: line 20 "`SETNX` a processing marker... onSend hook stores". Risk Assessment line 48 acknowledges store-after-commit but NOT the crash-leaves-processing-marker case. No owner re-check on replay stated.
- **Suggested fix:** Store the processing marker with a SHORT TTL (e.g. 30s) distinct from the 24h completed TTL so a crashed request self-heals; on completed-replay, assert stored owner == current `req.user.sub` before returning body (defense in depth even though key is scoped); refuse to enable idempotency on any route lacking `app.authenticate`.

## Finding 6: Webhook raw-body capture is a global-parser hazard; partial-route raw body is error-prone and unverified

- **Severity:** Medium
- **Location:** Phase 6, "Architecture → HMAC" + "Risk Assessment"
- **Flaw:** Verification is over RAW body (phase-06 line 27, 60) and the plan says "register raw-body for the webhook route" (line 27) and "configure raw body only for webhook route" (line 60). Fastify's default `application/json` content-type parser consumes and discards the raw buffer; capturing raw bytes for ONE route while every other route uses the JSON parser requires a content-type-parser or `onRequest`/`preParsing` shim. The plan gives no mechanism and no test that the bytes verified are byte-identical to what the provider signed (whitespace/key-ordering differences between `JSON.stringify` on send and the received buffer will silently break or, worse, be worked around by re-stringifying the parsed body — which reintroduces the parsed-vs-raw mismatch the plan warns about).
- **Failure scenario:** Implementer, hitting verification failures, "fixes" it by signing/verifying `JSON.stringify(req.body)` instead of the true raw bytes. Now any semantically-equal JSON passes regardless of byte form, and an attacker can canonicalize/reorder a forged payload to match — signature no longer binds the exact bytes. Alternatively a global raw-body parser is added, breaking TypeBox validation on all other routes.
- **Evidence:** `phase-06-...md:27,60`. No raw-body plugin (`@fastify/raw-body` or custom `addContentTypeParser`) named in "Related Code Files" (line 33-36) — only "`src/app.ts` (routes + raw body for webhook)" hand-waves it. App currently uses TypeBox validator globally (`src/app.ts:43`). No test asserts raw == signed bytes (phase-06 tests line 39 only says "tampered body/sig fails", not byte-fidelity).
- **Suggested fix:** Name the exact mechanism (`@fastify/raw-body` scoped via route `config.rawBody=true`, or a per-route `addContentTypeParser`); add a test that signs bytes B, sends B verbatim, and asserts verification passes while a re-serialized equivalent of B with reordered keys is NOT what's verified. Forbid re-stringify-then-verify in the phase text.

## Finding 7: Inventory "no over-sell" relies on atomic UPDATE but the saga consumer dedup + compensation can double-release stock

- **Severity:** Medium
- **Location:** Phase 4 "Risk Assessment" + Phase 6 "PaymentFailed" compensation
- **Flaw:** Reserve is atomic via `UPDATE ... WHERE stock_available >= q` (phase-04 line 20) — correct for over-sell on reserve. But the COMPENSATION/commit paths mutate stock without an equivalent guard:
  - PaymentSucceeded: `stock_reserved -= q` (commit) (phase-06 line 20). PaymentFailed: `stock_available += q, stock_reserved -= q` (phase-06 line 21).
  - Dedup for these consumers is `processed_messages` on eventId (phase-04 line 18), but the plan also adds Redis webhook dedup AND status guards as separate mechanisms (phase-06 line 61 "dedup webhook + outbox eventId"). If the webhook is delivered, processed, then the SAME payment outcome arrives via a second path (e.g. force endpoint + natural mock timer both fire — phase-06 line 28 default-succeed timer PLUS line 22 force endpoints can both enqueue an outcome), two DISTINCT eventIds carry the same logical "payment X failed" → both pass eventId dedup → stock released twice → `stock_available` inflated → over-sell on the next order.
- **Failure scenario:** Mock provider's default success timer fires AND an admin calls `/mock-payments/:id/fail` (or both succeed+fail race). Two different eventIds, both reference payment X. eventId dedup doesn't catch them (different ids). Compensation runs twice: `stock_available += q` applied twice. Inventory now shows phantom stock → subsequent orders over-sell real inventory.
- **Evidence:** Reserve guard `phase-04-...md:20`; commit/release have no `WHERE` guard quoted `phase-06-...md:20-21`. Dual outcome sources: default timer `phase-06-...md:28` + force endpoints `phase-06-...md:22`. Status guard mentioned `phase-06-...md:61` but keyed on order status, and the plan's own line 61 admits "double compensation (release twice)" is a live risk while relying on "status guard ... + dedup webhook + outbox eventId" — three uncoordinated mechanisms.
- **Suggested fix:** Make release/commit idempotent at the data layer: guard on payment/order status transition atomically (`UPDATE ... WHERE status = 'pending' RETURNING` — zero rows means already-transitioned, skip the stock mutation in the same tx). Dedup must key on the logical payment id + transition, not the transport eventId, so two different-eventId messages for the same outcome collapse to one effect. Add a test: succeed-event and fail-event for the same payment → exactly one terminal state, stock net-correct.

## Finding 8: `@fastify/rate-limit` Redis store + global limit gives no protection on the unauthenticated webhook → forgery/DoS amplification

- **Severity:** Medium
- **Location:** Phase 5 "Architecture → Rate-limit"; Phase 6 webhook route
- **Flaw:** Rate limit is global `max:100/min` today (`src/plugins/security.ts:10`) and phase-05 only swaps the store to Redis (phase-05 line 22) — it does NOT add a stricter per-route limit on `/webhooks/payment`. The webhook is unauthenticated by nature (called by the "provider"). With a shared global bucket keyed by IP, an attacker hammering forged webhooks either (a) burns the global limit affecting legit traffic, or (b) if the limit is per-IP and generous (100/min), gets 100 forgery attempts/min/IP to brute-force signatures or replay.
- **Failure scenario:** Attacker scripts forged webhook POSTs from a botnet; each IP gets 100/min. Combined with Finding 3 (weak secret) this is enough to offline-then-online confirm a guessed signature, or simply DoS the saga consumers with junk that still costs an HMAC verify + Redis lookup each.
- **Evidence:** `src/plugins/security.ts:10` single global `rateLimit({ max:100, timeWindow:'1 minute' })`; `phase-05-...md:22` only changes the store, no per-route override; webhook route has no rate-limit override in phase-06 (grep of phase-06 for "rate" → none).
- **Suggested fix:** Add a tight per-route rate limit on `/webhooks/payment` (e.g. low burst, keyed by IP) AND treat failed-HMAC as a fast-reject before any Redis/DB work. State this in phase-06.

## Cross-cutting note (not a separate finding)

The plan's dedup story is internally inconsistent: existing code uses a permanent Postgres PK (`processed_messages.messageId`, `src/infra/db/schema.ts:52`), phase-01 keeps that for consumers, but phase-06 introduces a parallel TTL'd Redis dedup for webhooks (`phase-06:29`) and phase-05 a third Redis idempotency layer. Three dedup mechanisms with different durability semantics is the root enabler of Findings 2 and 7. Recommend consolidating on durable (Postgres) dedup for anything that drives money/stock side effects; reserve Redis for the HTTP idempotency-key convenience layer only.
